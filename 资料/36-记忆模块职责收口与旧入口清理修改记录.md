# 36 记忆模块职责收口与旧入口清理修改记录

## 本次目标

本次修改完成 `memory_store.py`、`memory_repository.py`、`json_store.py` 三者职责冲突的最终清理。

前两步已经分别建立纯数据模型和统一业务 Service。本次让所有正式调用方使用新架构，并删除仅为过渡保留的旧名称和重复入口。

## 最终模块职责

### `memory_models.py`：数据模型层

只负责描述记忆数据：

```text
MemoryItem
LongTermMemory
默认记忆实例创建
旧 JSON 字段迁移
```

它不知道文件路径、Repository、Graph 和 CLI，也不执行新增或删除规则。

### `memory_repository.py`：持久化层

只负责将模型保存到后端或从后端恢复：

```text
MemoryRepository 协议
JsonMemoryRepository
LangGraphMemoryRepository
用户 namespace
JSON 文件路径映射
模型序列化和反序列化
```

Repository 按 `MemoryItem.id` 操作，不理解界面序号、去重规则和提示词格式。

### `memory_service.py`：业务层

只负责长期记忆的统一业务语义：

```text
空文本校验
文本规范化
重复检查
按用户可见序号删除
提示词格式化
调用 Repository 完成持久化
```

Graph 和 CLI 只调用 Service，不直接依赖具体 Repository。

### `storage/__init__.py`：依赖组装层

工厂函数负责把配置、namespace、Repository 和 Service 组装起来：

```python
def create_memory_service(settings) -> MemoryService:
    namespace = default_memory_namespace(settings)
    repository = JsonMemoryRepository(settings.memory_path, namespace)
    return LongTermMemoryService(repository, namespace)
```

这里是当前默认选择 JSON 后端的唯一位置。未来增加配置开关时，只需修改这个组装点。

### `json_session_store.py`：会话适配器

原来的 `storage/json_store.py` 同时出现 `JsonSessionStore` 和 `JsonMemoryStore`，名字宽泛且混合两个领域。

记忆业务迁出后，文件改名为 `json_session_store.py`，现在只处理会话存储适配。文件名和实际职责一致。

## 删除了什么

### 1. 删除 `memory_store.py`

该文件原先包含模型、文件操作、业务增删和提示词格式化。经过前两步迁移后已经没有独立职责，因此整体删除。

继续保留它作为兼容入口会造成两个问题：

1. 新开发者仍可能从旧文件导入并绕开 Service。
2. 同一个概念存在新旧两个入口，架构边界只停留在文档上。

项目尚处于早期阶段，当前清理旧 API 的成本较低，因此选择彻底删除而不是长期维护别名。

### 2. 删除 `JsonMemoryStore`

`JsonMemoryStore` 的业务能力已经属于 `LongTermMemoryService`。工厂现在直接组装 Service 和 JSON Repository，不再使用带具体存储技术名称的业务类。

### 3. 删除 `create_memory_store`

正式入口改为：

```python
create_memory_service(settings)
```

名称明确告诉调用者得到的是业务服务，而不是底层存储对象。

## 调用方如何变化

Graph 中：

```python
memory_service = create_memory_service(settings)
memory_service.format_for_prompt()
memory_service.add(...)
```

CLI 中：

```python
memory_service = create_memory_service(settings)
memory_service.add(...)
memory_service.delete(...)
```

Graph 和 CLI 都不再出现 JSON 相关类名。以后底层后端切换时，这两个业务调用方无需修改。

## 最终依赖方向

```text
Graph / CLI
    |
    v
MemoryService
    |
    v
MemoryRepository
    |
    +-- JsonMemoryRepository
    +-- LangGraphMemoryRepository

MemoryService 和 MemoryRepository
    |
    v
MemoryItem / LongTermMemory
```

依赖只从使用者指向被使用者，不再出现 Repository 回头调用一个包含业务逻辑的 Store 文件。

## 测试如何重新分层

旧的 `test_memory_store.py` 同时测试模型兼容、业务增删和提示词格式化，测试结构复制了原来的职责混合。

现在拆为：

```text
test_memory_models.py：旧字段迁移、默认实例隔离
test_memory_repository.py：JSON 创建、往返读写、namespace 路径安全
test_memory_service.py：去重、删除和提示词业务规则
```

测试文件与生产模块一一对应，某层出现失败时更容易定位原因。

## 对成熟商业项目的意义

### 1. 降低修改影响范围

业务规则变化只修改 Service，JSON 格式变化只修改 Repository，字段变化只修改 Model。

### 2. 存储后端可替换

JSON 是当前默认选择，但 Graph 和 CLI 已不依赖它。接入真正的 LangGraph Store 时，主要改动集中在工厂组装和对应 Repository。

### 3. 防止绕过业务规则

删除旧辅助函数后，正式调用方不能再直接修改模型列表并自行保存，长期记忆写入统一经过 Service。

### 4. 提高可测试性

Service 测试使用内存 Repository，Repository 测试使用临时目录。测试不再互相依赖，执行更快、失败原因更明确。

### 5. 命名成为架构约束

文件名和类名不只是说明文字，也会影响开发者把新代码放在哪里。准确命名可以减少职责再次混合的概率。
