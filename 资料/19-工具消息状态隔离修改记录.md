# 19 工具消息状态隔离修改记录

## 本次目标

本次是“完整工具调用框架化”的第三步：将工具调用过程消息隔离为“本轮临时状态”，避免工具消息随着 LangGraph checkpointer 跨轮累积。

HumanChat 现在已经接入了 checkpointer。checkpointer 会保存 Graph 状态，这对短期对话记忆是好事，但工具调用消息和普通聊天消息的性质不同：

```text
普通聊天消息：用户和助手真正看见的对话历史，应该保留
工具调用消息：Agent 为了本轮推理临时产生的中间过程，不一定应该跨轮保留
```

所以本次要把 `tool_messages` 明确设计为本轮工具 scratchpad。

## 修改前的问题

修改前 `ChatState` 中：

```python
tool_messages: Annotated[list[Any], add_messages] = Field(default_factory=list)
```

`add_messages` 的行为是追加消息。

这对 `messages` 是合适的，因为对话历史确实需要持续追加。

但对 `tool_messages` 来说会有风险：

```text
第一轮问题调用工具，产生工具结果 A
第二轮问题调用工具，旧工具结果 A 仍然可能留在状态中
最终 prompt 可能混入过期工具上下文
```

这会带来几个问题：

```text
模型可能参考旧工具结果回答新问题
token 成本越来越高
debug 输出越来越嘈杂
工具调用过程和用户对话历史边界不清
```

## 修改了什么

### 1. `tool_messages` 不再使用 `add_messages`

修改前：

```python
tool_messages: Annotated[list[Any], add_messages] = Field(default_factory=list)
```

修改后：

```python
tool_messages: list[Any] = Field(default_factory=list)
```

这样 `tool_messages` 不再作为自动追加的消息通道，而是由节点显式维护完整列表。

### 2. 每轮对话开始时清空工具消息

在 `prepare_context` 中新增：

```python
"tool_messages": []
```

每次用户提出新问题，Graph 都会清空上一轮工具 scratchpad。

这样可以确保：

```text
普通对话历史继续保存
工具中间过程每轮重新开始
```

### 3. `call_agent_model` 显式返回完整工具对话

因为 `tool_messages` 不再自动追加，所以模型节点需要显式返回完整的工具对话：

```python
return {"tool_messages": [*conversation, response]}
```

这保证当前工具循环仍然能保留：

```text
当前问题
模型的工具调用请求
工具返回结果
模型下一次判断
```

### 4. `execute_project_tools` 显式合并工具结果

`ToolNode` 执行后只返回新的工具结果消息。

因此本次将它和已有本轮工具消息合并：

```python
tool_result_messages = result.get("tool_messages", [])
return {
    "tool_messages": [*state.tool_messages, *tool_result_messages],
    "tool_call_count": state.tool_call_count + 1,
}
```

这样既能保留本轮工具循环上下文，又不会污染下一轮用户问题。

## 为什么这样做

### 1. 工具消息不是用户对话历史

工具消息通常包含：

```text
模型请求调用哪个工具
工具调用参数
工具返回的原始结果
中间观察信息
```

这些内容对 Agent 当前推理有用，但不一定适合长期保留在对话历史里。

用户真正关心的是：

```text
用户问了什么
助手最终答了什么
```

所以 `messages` 和 `tool_messages` 应该分开治理。

### 2. 避免过期上下文污染

商业 Agent 最怕“看起来回答很自信，但参考的是旧上下文”。

如果旧工具结果被带入新问题，模型可能会：

```text
引用已经不相关的文件内容
把上一个问题的搜索结果当成当前问题依据
因为上下文混乱而做出错误判断
```

清空本轮工具 scratchpad 可以显著降低这种风险。

### 3. 控制 token 成本

如果工具消息持续追加，长时间使用后 prompt 会越来越大。

这会导致：

```text
响应变慢
成本上升
上下文窗口被无关工具结果占用
模型注意力被稀释
```

本轮隔离可以让工具上下文保持短小、相关、可控。

### 4. 让状态语义更清楚

修改后状态语义更明确：

```text
messages：跨轮保留的用户/助手对话历史
tool_messages：当前问题内部的工具调用 scratchpad
memory_prompt：当前问题读取出的长期记忆提示
assistant_text：最终助手回复
```

这种边界对后续接 MCP、streaming、tracing、Web UI 都更友好。

## 对商业项目的意义

### 1. 更稳定的回答质量

工具结果只服务于当前问题，减少旧结果干扰。

### 2. 更低的运行成本

工具上下文不会无限增长，减少不必要的 token 消耗。

### 3. 更清晰的审计边界

用户可见历史和 Agent 内部工具过程分离。

这对于未来做：

```text
会话回放
工具调用审计
隐私数据处理
用户导出聊天记录
```

都更安全。

### 4. 更适合未来 MCP 工具

MCP 工具结果可能更长、更敏感、更复杂。

如果 MCP 工具结果也跨轮累积，风险会比本地文件工具更高。

因此在真正接 MCP 前，先把工具状态隔离做好，是更稳妥的工程顺序。
