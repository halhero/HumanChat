# 08 存储层升级与数据模型统一

## 当前状态

HumanChat 当前使用多个文件保存数据：

```text
data/sessions/*.json
data/memory/user_profile.json
characters/*.yaml
```

这种方式适合原型阶段，优点是简单、直观、容易调试。

但随着功能增加，会出现一些问题：

```text
查询历史会话不方便
消息和记忆没有统一 ID
无法方便统计、搜索、分页
未来 UI 展示会变麻烦
多个 Store 的接口风格不完全统一
```

## 要做什么

本阶段不是立刻把所有 JSON 删除，也不是立刻强行迁移数据库。

更稳妥的做法是先建立统一数据访问层，再决定是否迁移到 SQLite。

需要整理这些概念：

```text
Session：一次会话
Message：会话中的一条消息
Memory：长期记忆项
Character：角色配置
```

然后为它们建立更统一的 Store 接口。

例如：

```text
SessionStore
MemoryStore
CharacterStore
```

上层代码不再直接关心 JSON 文件路径，而是调用 Store。

## 为什么这么做

成熟项目中，数据层应该是清晰的。

如果业务代码到处直接读写 JSON，后续会遇到几个问题：

```text
改存储格式会影响很多文件
无法轻松迁移到数据库
无法统一做校验和备份
UI 很难复用数据查询逻辑
测试也会更难写
```

统一数据访问层的意义是：

```text
先把接口稳定下来，再决定底层用 JSON 还是 SQLite
```

这比一上来重构数据库更安全。

## 怎么做

第一步可以新增目录：

```text
human_chat/storage/
```

里面放：

```text
base.py
json_session_store.py
json_memory_store.py
```

示例接口：

```python
class SessionStore:
    def create(self) -> dict:
        ...

    def load(self, session_id: str) -> dict:
        ...

    def save(self, session: dict) -> None:
        ...

    def list_recent(self, limit: int = 10) -> list[dict]:
        ...
```

MemoryStore 示例：

```python
class MemoryStore:
    def load(self) -> LongTermMemory:
        ...

    def save(self, memory: LongTermMemory) -> None:
        ...

    def add(self, category: str, text: str) -> bool:
        ...

    def delete(self, category: str, index: int) -> str | None:
        ...
```

等接口稳定后，再考虑：

```text
SQLiteSessionStore
SQLiteMemoryStore
```

并提供迁移脚本：

```text
JSON -> SQLite
SQLite -> JSON 导出
```

## 完成标准

这一阶段完成后，应具备：

```text
上层 runtime/cli 不直接读写 session JSON
memory 操作通过统一 Store 完成
现有 JSON 数据仍然兼容
后续可平滑迁移 SQLite
```
