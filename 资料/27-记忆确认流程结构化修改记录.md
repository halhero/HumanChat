# 27 记忆确认流程结构化修改记录

## 本次目标

本次是“记忆模块 LangChain / LangGraph 框架化”的第六步：把候选长期记忆的确认流程结构化。

前一步已经把自动记忆提取移入 Graph。现在 Graph 会返回：

```python
result["memory_candidates"]
```

但 CLI 之前直接处理裸 `list[dict]`：

```python
for candidate in candidates:
    category = candidate.get("category", "note")
    text = candidate.get("text", "").strip()
```

这种方式适合早期原型，但不适合成熟项目。

记忆确认是一个重要的人类确认流程，未来可能由 CLI、Web UI、LangGraph interrupt、甚至移动端共同处理。因此它需要一个稳定的请求模型。

## 修改了什么

### 1. 新增 `human_chat/memory_review.py`

新增模块用于描述候选记忆和确认请求。

目前包含：

```python
class MemoryCandidate(BaseModel):
    category: str
    text: str


class MemoryExtractionResult(BaseModel):
    candidates: list[MemoryCandidate]


class MemoryReviewRequest(BaseModel):
    candidates: list[MemoryCandidate] = Field(default_factory=list)
    require_confirmation: bool = True
```

字段含义：

```text
candidates：等待用户审查的候选长期记忆
require_confirmation：是否必须经过用户确认
```

当前项目仍然坚持“候选记忆必须用户确认后才保存”，所以默认：

```python
require_confirmation = True
```

同时，`memory_extractor.py` 不再自己定义候选记忆模型，而是从 `memory_review.py` 导入 `MemoryCandidate` 和 `MemoryExtractionResult`。

这样候选记忆的数据结构只有一份定义，提取节点和确认流程使用同一个模型。

### 2. 新增 `create_memory_review_request`

新增函数：

```python
def create_memory_review_request(candidates: list[dict | MemoryCandidate]) -> MemoryReviewRequest:
    ...
```

它负责把 Graph 返回的候选数据规范化为 `MemoryReviewRequest`。

处理规则：

```text
dict 会转换为 MemoryCandidate
text 会 trim
category 会 trim 并转小写
空 text 会被过滤
空 category 会回退为 note
```

这样 CLI 不再直接信任裸 dict，而是先得到结构化 review request。

### 3. CLI 使用结构化确认请求

修改前：

```python
def _confirm_memory_candidates(settings, candidates):
    if not candidates:
        return

    for candidate in candidates:
        category = candidate.get("category", "note")
        text = candidate.get("text", "").strip()
```

修改后：

```python
review_request = create_memory_review_request(candidates)
if not review_request.candidates:
    return

for candidate in review_request.candidates:
    category = candidate.category
    text = candidate.text
```

CLI 的交互体验保持不变：

```text
发现候选长期记忆：
1. preference: ...
保存这条记忆？y/N：
```

但底层数据结构更清晰了。

## 为什么这样做

### 1. 长期记忆写入必须谨慎

长期记忆会影响未来所有对话。

如果模型误提取、误分类、误保存，就可能长期污染 Agent 行为。

因此成熟 Agent 不应该静默写入长期记忆，而应该保留人类确认流程。

### 2. CLI 不应该处理裸 dict

裸 dict 的问题是：

```text
字段可能缺失
字段类型可能不对
text 可能为空
category 可能大小写不统一
未来增加字段时没有统一入口
```

结构化模型可以把这些规范化逻辑集中起来。

### 3. 为 LangGraph interrupt 做准备

LangGraph 的 human-in-the-loop / interrupt 通常需要一个明确的请求对象。

未来可以把当前的：

```python
MemoryReviewRequest
```

直接作为 interrupt payload 的基础。

例如未来流程可以变成：

```text
extract_memory
  -> interrupt(MemoryReviewRequest)
  -> save_approved_memory
```

当前先结构化请求，后续接 interrupt 时迁移成本更低。

### 4. 为 Web UI 做准备

如果未来 HumanChat 做 Web UI，前端需要拿到一个稳定结构：

```json
{
  "candidates": [
    {"category": "preference", "text": "..."}
  ],
  "require_confirmation": true
}
```

而不是让 UI 自己猜测 Graph 返回的裸 dict 应该怎么处理。

## 对成熟项目的意义

### 1. 用户控制权更清晰

记忆候选和记忆写入之间有明确 review request。

### 2. 交互层更可替换

CLI、Web UI、interrupt 可以复用同一种确认请求。

### 3. 数据更稳定

候选记忆在进入确认流程前会被规范化。

### 4. 后续扩展更自然

未来可以在 `MemoryReviewRequest` 中继续增加：

```text
request_id
created_at
source_turn_id
allow_edit
review_decisions
```
