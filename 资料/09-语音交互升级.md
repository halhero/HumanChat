# 09 语音交互升级

## 当前状态

HumanChat 现在已经有：

```text
TTS 语音输出
音频文件 STT 输入
文字输入模式
```

但目前语音输入方式还比较原始：用户需要输入音频文件路径。

这说明语音链路已经打通，但还不是自然的语音聊天体验。

## 要做什么

本阶段要把语音输入从“音频文件输入”升级到更接近真实语音对话的体验。

可以分为几个能力：

```text
麦克风录音
录音开始 / 结束控制
STT 识别
识别文本展示
确认发送 / 重录
TTS 播放开关
会话中切换输入模式
```

第一版不需要实时流式识别，先做可控录音即可。

## 为什么这么做

项目目前已经有 GPT-SoVITS 语音输出。如果输入仍然主要靠打字，那么它更像“带语音播报的聊天程序”。

要成为真正的语音 Agent，需要形成闭环：

```text
用户说话
  -> STT 转文字
  -> Agent 理解并回复
  -> TTS 播放
```

同时，语音交互必须可控。用户需要看到识别结果，否则 STT 识别错了会直接进入 Agent，体验会很差。

所以建议保留确认步骤：

```text
识别结果：……
发送？ y/n/r
```

其中：

```text
y：发送
n：取消
r：重新录音
```

## 怎么做

可以新增：

```text
human_chat/audio_recorder.py
```

职责：

```text
录音
保存临时 wav 文件
返回音频路径
```

输入层可以扩展：

```python
class MicrophoneInputProvider:
    def read_question(self) -> str:
        audio_path = record_audio(...)
        text = transcribe_audio_file(...)
        confirm = input("发送？")
        ...
```

CLI 启动时输入模式可以变成：

```text
1. 文字输入
2. 音频文件输入
3. 麦克风输入
```

同时可以增加运行时命令：

```text
/input text
/input audio-file
/input mic
/tts on
/tts off
```

需要注意：

```text
录音依赖可能需要额外安装
Windows 麦克风权限可能导致失败
失败时应回退到文字输入
临时音频文件应保存到 ignored 目录
```

## 完成标准

这一阶段完成后，应具备：

```text
可以从麦克风录音
可以识别录音为文字
发送前可以确认或重录
可以在会话中切换输入模式
TTS 可以开关
音频失败不影响文字聊天
```
