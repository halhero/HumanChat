# 23 长期记忆数据模型结构化修改记录

## 本次目标

本次是“记忆模块 LangChain / LangGraph 框架化”的第二步：把长期记忆从简单字符串列表升级为结构化记忆项。

修改前，长期记忆格式是：

```json
{
  "preferences": [],
  "facts": [],
  "notes": []
}
```

这个格式直观、适合原型阶段，但它缺少成熟项目需要的元数据。

成熟记忆系统需要知道：

```text
这条记忆的唯一 ID 是什么
它属于什么类别
它什么时候创建
它什么时候更新
它来自手动添加、自动提取，还是默认模板
它的置信度是多少
```

所以本次引入 `MemoryItem`，让每条记忆成为可追踪、可审计、可迁移的数据项。

## 修改了什么

### 1. 新增 `MemoryItem`

在 `human_chat/memory_store.py` 中新增：

```python
class MemoryItem(BaseModel):
    id: str
    category: str
    text: str
    created_at: str
    updated_at: str
    source: str = "manual"
    confidence: float | None = None
```

字段含义：

```text
id：记忆项唯一标识
category：记忆类别，当前使用 preferences / facts / notes
text：记忆正文
created_at：创建时间
updated_at：更新时间
source：来源，例如 manual / default / legacy / example
confidence：置信度，当前预留给自动记忆提取使用
```

### 2. `LongTermMemory` 改为持有 `items`

修改后：

```python
class LongTermMemory(BaseModel):
    items: list[MemoryItem]
```

这意味着长期记忆的真实存储结构不再是三个字符串数组，而是一组结构化记忆项。

### 3. 保留兼容属性

为了不破坏现有代码和使用习惯，`LongTermMemory` 仍然提供：

```python
memory.preferences
memory.facts
memory.notes
```

但它们现在是从 `items` 中按 category 计算出来的只读文本列表。

这样当前测试、CLI 展示、prompt 格式化逻辑仍然可以继续使用熟悉的分类视角。

### 4. 兼容旧 JSON 格式

`LongTermMemory.__init__` 会接受旧格式：

```python
LongTermMemory(
    preferences=["..."],
    facts=["..."],
    notes=["..."],
)
```

并自动转换为 `MemoryItem`。

旧数据会被标记为：

```text
source = "legacy"
```

这意味着用户已有的 `data/memory/user_profile.json` 即使还是旧格式，也可以继续读取。

### 5. 添加记忆时生成结构化项

`add_memory_item()` 不再向字符串列表 append，而是创建：

```python
MemoryItem(
    category=normalized_category,
    text=normalized,
    source="manual",
)
```

这样用户通过 `/memory add` 添加的记忆会带上 ID、时间和来源。

### 6. 删除记忆时按分类序号兼容旧体验

`delete_memory_item()` 仍然支持：

```text
/memory delete preference 1
```

内部会先找到对应分类下的第 N 条记忆，然后根据 `id` 从 `memory.items` 中删除。

也就是说，底层模型升级了，但用户体验没有突然变化。

### 7. 更新示例记忆文件

`data/memory/user_profile.example.json` 已更新为结构化 `items` 格式。

它可以作为新用户记忆文件的参考模板。

## 为什么这样做

### 1. 字符串列表不适合成熟记忆系统

简单字符串列表无法回答这些问题：

```text
这条记忆是什么时候产生的？
它是用户手动添加的，还是模型自动提取的？
它是否来自旧数据迁移？
它能不能被单独编辑？
它是否可以被稳定引用？
```

这些问题在商业项目中非常重要。

### 2. 每条记忆都应该有 ID

后续如果要做 UI、编辑、审计、删除、同步、迁移，就不能只靠“分类中的第几个”来定位记忆。

序号适合 CLI 操作，但不适合内部系统。

所以本次引入 `id`，让每条记忆可以被稳定引用。

### 3. 来源信息很关键

长期记忆可能来自：

```text
用户手动添加
模型自动提取
默认项目模板
旧数据迁移
外部导入
```

这些来源的可信度不同。

例如，用户手动添加的记忆通常可信度更高；模型自动提取的记忆可能需要用户确认。

因此 `source` 是后续记忆治理的重要字段。

### 4. 为自动记忆提取和确认流程做准备

后续步骤会把自动记忆提取移入 Graph，并结构化确认流程。

如果长期记忆仍然只是字符串，自动提取出来的候选记忆很难记录：

```text
来源
置信度
创建时间
确认状态
```

所以模型升级必须先做。

## 对成熟项目的意义

### 1. 可审计

每条记忆都有 ID、来源和时间。

### 2. 可迁移

未来从 JSON 迁移到 LangGraph Store 或 SQLite 时，可以逐条迁移，而不是迁移一个大字符串集合。

### 3. 可扩展

后续可以继续为 `MemoryItem` 增加：

```text
scope
namespace
owner_id
project_id
last_used_at
usage_count
```

### 4. 用户体验保持稳定

虽然底层模型升级了，但 `/memory add`、`/memory delete`、prompt 展示仍然可以按原来的分类方式工作。

