# 06 自动长期记忆提取

## 当前状态

HumanChat 现在已经有长期记忆文件：

```text
data/memory/user_profile.json
```

并且已经支持在聊天中使用命令手动维护：

```text
/memory
/memory add preference ...
/memory add fact ...
/memory delete ...
```

同时，`human_chat/graph.py` 会在每轮对话前读取长期记忆，并把它注入系统提示词。

也就是说，长期记忆现在已经可以被使用，也可以被手动修改。但它还不能自动从对话中发现新的长期信息。

## 要做什么

本阶段要让 Agent 在每轮对话后，尝试从用户输入和助手回复中提取“候选长期记忆”。

候选记忆需要分为三类：

```text
preferences：用户偏好，比如沟通方式、代码修改方式、输出风格
facts：长期事实，比如项目技术栈、项目目标、用户正在做什么
notes：其他值得保留但不适合归类为偏好或事实的备注
```

第一版不要静默自动写入长期记忆，而是先展示给用户确认。

例如：

```text
发现候选长期记忆：
1. preference: 用户希望修改代码前先看到方案说明。
是否保存？ y/n
```

用户确认后，才写入 `data/memory/user_profile.json`。

## 为什么这么做

成熟 Agent 不应该只在当前 session 里理解用户，也应该跨 session 保留稳定偏好和项目背景。

但长期记忆是敏感能力。如果模型误解用户、记错事实、保存了临时情绪或不重要信息，会污染后续所有对话。

所以第一版必须遵循一个原则：

```text
模型可以提出候选记忆，但最终写入必须由用户确认。
```

这样既能让记忆逐渐增长，又不会失控。

## 怎么做

建议新增模块：

```text
human_chat/memory_extractor.py
```

职责：

```text
接收本轮 user_text 和 assistant_text
调用 LLM 提取候选记忆
返回结构化候选结果
```

示例结构：

```python
class MemoryCandidate(BaseModel):
    category: str
    text: str


class MemoryExtractionResult(BaseModel):
    candidates: list[MemoryCandidate] = Field(default_factory=list)
```

示例函数：

```python
def extract_memory_candidates(llm, user_text: str, assistant_text: str) -> MemoryExtractionResult:
    ...
```

在 `ChatRuntime.ask()` 或 Graph 的后置阶段中接入：

```text
用户输入
  -> Agent 回复
  -> 提取候选记忆
  -> CLI 展示候选
  -> 用户确认
  -> 写入 memory_store
```

需要注意：

```text
不要把所有对话都保存为长期记忆
不要保存明显临时的信息
不要保存隐私信息，除非用户明确表达希望记住
先做精确去重，再考虑语义去重
```

## 完成标准

这一阶段完成后，应具备：

```text
每轮对话后能生成候选长期记忆
候选记忆按 preference / fact / note 分类
用户确认后才写入
重复文本不会重复保存
用户可以继续通过 /memory 查看和删除
```
