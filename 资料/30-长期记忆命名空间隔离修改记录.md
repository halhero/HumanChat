# 30 长期记忆命名空间隔离修改记录

## 本次目标

本次目标是让长期记忆的 namespace 不再只是语义概念，而是实际影响 JSON 文件存储位置。

之前项目已经有：

```python
("users", settings.memory_user_id, "memory")
```

但 JSON Repository 仍然只写入同一个 `memory_path`。

这会造成一个实际问题：

```text
设置不同 HUMANCHAT_MEMORY_USER_ID
代码语义上像是不同用户
实际仍可能写入同一个 JSON 文件
```

成熟项目不能出现这种“看起来隔离、实际上未隔离”的数据边界。

## 修改了什么

### 1. 新增 `memory_path_for_namespace`

`human_chat/memory_repository.py` 新增：

```python
def memory_path_for_namespace(base_path: Path, namespace: MemoryNamespace) -> Path:
    ...
```

默认 namespace：

```python
("users", "default", "memory")
```

继续使用原路径：

```text
data/memory/user_profile.json
```

非默认用户会使用派生路径：

```text
data/memory/user_profile.alice.json
data/memory/user_profile.bob.json
```

### 2. JSON Repository 初始化时解析真实路径

修改前：

```python
self.path = path
```

修改后：

```python
self.path = memory_path_for_namespace(path, namespace)
```

也就是说，同一个 base path 会根据 namespace 映射到不同文件。

### 3. 用户 ID 做路径安全处理

新增：

```python
def _safe_path_segment(value: str) -> str:
    ...
```

它会把不适合文件名的字符转换为 `_`，避免用户 ID 直接污染文件路径。

## 为什么这样做

### 1. 多用户隔离必须是真实行为

长期记忆属于用户数据。

如果不同用户写进同一个文件，会产生严重问题：

```text
用户 A 看到用户 B 的偏好
用户 B 的记忆污染用户 A 的回答
删除记忆时误删其他用户数据
```

所以 namespace 必须落实到存储边界。

### 2. 保持默认用户兼容

默认用户继续使用原文件：

```text
user_profile.json
```

这样已有本地记忆不会因为本次修改突然换位置。

### 3. JSON 后端仍适合当前阶段

当前项目还处在功能完善阶段。

直接引入数据库会增加部署成本。

先把 JSON 后端做成真实 namespace 隔离，可以在不破坏当前体验的前提下提升架构质量。

## 怎么做

核心规则：

```python
if namespace == ("users", "default", "memory"):
    return base_path
```

非默认用户：

```python
return base_path.with_name(f"{base_path.stem}.{user_id}{base_path.suffix}")
```

例如：

```text
base_path = data/memory/user_profile.json
memory_user_id = alice
实际路径 = data/memory/user_profile.alice.json
```

## 对成熟项目的意义

### 1. 降低数据串扰风险

长期记忆直接影响未来回复，因此隔离边界必须可靠。

### 2. 为数据库迁移保留 namespace 语义

未来迁移到 SQLite、Postgres 或 LangGraph Store 时，可以继续沿用 namespace：

```text
namespace + key + value
```

### 3. 支持更多记忆作用域

后续可以扩展：

```text
("users", "alice", "memory")
("projects", "HumanChat", "memory")
("characters", "nanami", "memory")
```
