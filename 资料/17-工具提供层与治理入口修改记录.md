# 17 工具提供层与治理入口修改记录

## 本次目标

本次是“完整工具调用框架化”的第一步：把工具来源从 `graph.py` 中抽离出来，建立统一的工具提供层。

这里要特别说明：本次不是要把项目固定为“本地工具项目”，也不是否认 MCP 的价值。成熟 Agent 项目的工具来源通常是混合的：

```text
本地 Python 工具
MCP 服务器工具
HTTP API 工具
数据库工具
搜索工具
内部业务系统工具
```

所以本次真正要建立的是一个统一入口：

```text
Local tools / MCP tools / HTTP tools
        ↓
ToolProvider
        ↓
LangGraph ToolNode
        ↓
Agent Graph
```

当前项目还没有真正接 MCP，因此这一步先把已有本地项目工具接入 `ToolProvider`，但结构上为后续 MCP 工具接入预留位置。

## 修改了什么

### 1. 新增 `human_chat/tool_provider.py`

新增工具提供层文件，包含：

```python
ToolMetadata
ToolProvider
LocalProjectToolProvider
CompositeToolProvider
create_tool_provider
```

这些对象的职责如下：

```text
ToolMetadata：描述工具的治理信息
ToolProvider：定义工具提供者协议
LocalProjectToolProvider：提供当前项目已有的本地只读工具
CompositeToolProvider：组合多个工具来源
create_tool_provider：给 graph.py 使用的统一创建入口
```

### 2. `graph.py` 不再直接依赖本地工具函数

修改前：

```python
from human_chat.tools import get_project_tools

project_tools = get_project_tools()
```

修改后：

```python
from human_chat.tool_provider import create_tool_provider

tool_provider = create_tool_provider(settings)
project_tools = tool_provider.get_tools()
```

这样 `graph.py` 只关心“当前 Agent 可以使用哪些工具”，不再关心这些工具来自本地函数、MCP 服务器，还是未来的业务 API。

### 3. 显式稳定工具名称

原先工具名来自 Python 函数名：

```text
list_project_files_tool
read_project_file_tool
search_project_text_tool
```

本次改为显式工具名：

```python
@tool("list_project_files")
@tool("read_project_file")
@tool("search_project_text")
```

这样做后，工具名变成稳定接口，不再受 Python 函数重命名影响。

### 4. 增加工具治理元数据

`ToolMetadata` 中记录：

```python
name: str
description: str
source: str
read_only: bool
requires_confirmation: bool
```

这些字段目前还没有全部用于运行时策略，但它们是商业项目必要的治理入口。

未来可以基于这些信息做：

```text
只允许某些角色使用某些工具
对非只读工具增加用户确认
记录不同来源工具的调用情况
在 UI 中展示 Agent 当前可用工具
接 MCP 后区分 local_project / mcp / http_api 来源
```

### 5. 防止工具名重复

`CompositeToolProvider` 初始化时会检查所有工具元数据名称是否重复。

如果后续 MCP 工具和本地工具出现同名，就会在启动阶段尽早失败，而不是等模型调用时才出现难以排查的问题。

## 为什么这样做

### 1. 成熟项目不能让 `graph.py` 直接拼工具列表

早期代码中：

```python
project_tools = get_project_tools()
```

看起来很简单，但随着工具来源变多，很容易变成：

```python
project_tools = get_project_tools()
mcp_tools = get_mcp_tools()
http_tools = get_http_tools()
tools = project_tools + mcp_tools + http_tools
```

然后权限、日志、禁用工具、环境区分都会继续堆在 `graph.py` 里。

成熟项目里，Graph 应该专注流程编排，不应该承担工具来源管理职责。

### 2. MCP 需要接入点，而不是直接塞进 Graph

MCP 工具确实是成熟 Agent 项目的重要方向。但即使用 MCP，也需要决定：

```text
哪些 MCP 工具允许暴露给模型
哪些 MCP 工具需要确认
哪些 MCP 工具只能在开发环境启用
MCP 工具名是否和本地工具冲突
MCP 工具调用如何记录日志
```

这些都不应该直接写在 `graph.py` 中。

所以先建立 `ToolProvider`，是为了让 MCP 将来接入时有一个正确位置。

### 3. 工具名是系统契约

模型调用工具时依赖工具名。日志、测试、权限策略、评估集也会依赖工具名。

如果工具名跟着 Python 函数名变化，普通代码重构就可能破坏 Agent 行为。

显式工具名可以让工具接口更稳定。

### 4. 工具治理是商业项目的安全基础

商业 Agent 项目中的工具可能访问文件、数据库、外部服务，甚至执行真实业务动作。

因此系统必须回答：

```text
当前 Agent 能使用哪些工具？
工具来自哪里？
工具是否只读？
工具是否需要用户确认？
是否存在同名工具冲突？
```

本次修改就是为这些问题建立基础结构。

## 当前完成程度

本次完成了：

```text
工具来源从 graph.py 抽离
新增 ToolProvider 统一入口
保留当前本地工具能力
为 MCP / HTTP 工具预留扩展位置
显式稳定当前工具名称
增加工具元数据
增加重复工具名检查
```

