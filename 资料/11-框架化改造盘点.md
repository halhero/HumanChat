# 11 框架化改造盘点

## 背景

HumanChat 当前已经使用了 LangGraph 的 `StateGraph`，也使用了 LangChain 的 prompt、message、structured output 等能力。

但是从整体代码看，项目里仍然有不少 Agent 基础设施是我们自己实现的，例如：

```text
工具决策
工具执行
短期会话消息保存
长期记忆存储
人类确认
运行时编排
```

这些实现适合早期原型阶段，因为它们直观、容易解释、改动成本低。但随着项目逐渐成熟，继续大量手写这些能力会带来问题：

```text
和 LangGraph / LangChain 的生态脱节
状态流转越来越分散
工具调用缺少标准 schema
记忆与会话持久化难以扩展
后续接入 MCP、UI、可观测性会更吃力
```

所以后续需要逐步把“Agent 基础能力”迁移回 LangGraph / LangChain 的标准能力上。

下面分点列出当前所有值得框架化的部分。

---

## 1. 工具定义与工具调用框架化

### 当前实现

相关文件：

```text
human_chat/tools.py
human_chat/graph.py
human_chat/cli.py
```

当前工具是普通 Python 函数：

```python
list_project_files(...)
read_project_file(...)
search_project_text(...)
```

CLI 可以通过命令手动调用：

```text
/files
/read
/search
```

Graph 中又额外手写了：

```text
TOOL_DECISION_PROMPT
ToolDecision
decide_tool_use
execute_tool
route_after_tool_decision
```

也就是说，现在工具调用流程是我们自己实现的。

### 应该使用的框架能力

LangChain / LangGraph 提供了标准工具机制：

```text
@tool
StructuredTool
ToolNode
模型 tool calling
```

工具可以带有：

```text
名称
描述
参数 schema
类型注解
错误处理
运行时上下文
```

### 为什么要改

当前手写工具调用的问题：

```text
工具 schema 不够标准
模型输出 tool_name / arguments 容易不稳定
execute_tool 中 if/else 会随着工具增加变臃肿
后续接 MCP 工具时不自然
无法充分利用 LangChain 的工具调用生态
```

使用框架工具后：

```text
工具描述可以直接暴露给模型
模型决定是否调用工具
ToolNode 统一执行工具
新增工具更容易
未来接 MCP 更自然
```

### 怎么改

建议新增或改造：

```text
human_chat/tools.py
human_chat/tool_registry.py
human_chat/graph.py
```

把工具改成类似：

```python
from langchain_core.tools import tool

@tool
def read_project_file_tool(path: str) -> str:
    """Read a UTF-8 text file inside the HumanChat project."""
    ...
```

Graph 中移除：

```text
ToolDecision
decide_tool_use
execute_tool
```

改用：

```text
llm.bind_tools(tools)
ToolNode(tools)
conditional edge 判断是否有 tool_calls
```

### 优先级

高。

这是当前最明显的“重复造轮子”部分，应优先框架化。

---

## 2. 短期会话记忆框架化

### 当前实现

相关文件：

```text
human_chat/runtime.py
human_chat/session_store.py
human_chat/storage/json_store.py
human_chat/cli.py
```

当前短期会话记忆由 `ChatRuntime` 手动维护：

```python
self.messages
session["messages"]
save_session(...)
```

每次调用 Graph 时传入：

```python
{
    "question": question,
    "messages": self.messages,
}
```

Graph 返回后再手动保存 messages。

### 应该使用的框架能力

LangGraph 提供 checkpoint / checkpointer 机制。

典型方式是：

```python
graph.compile(checkpointer=...)
app.invoke(
    input,
    config={"configurable": {"thread_id": session_id}},
)
```

短期消息状态应由 LangGraph 根据 `thread_id` 持久化。

### 为什么要改

当前手动保存短期消息的问题：

```text
Runtime 和 Graph 状态耦合
会话恢复逻辑在框架外
未来支持 streaming / interrupt / tool_calls 会更困难
无法使用 LangGraph 原生持久执行能力
```

使用 checkpointer 后：

```text
会话状态由 Graph 管理
thread_id 对应 session
human-in-the-loop 可以安全暂停/恢复
工具调用和消息状态保持一致
```

### 怎么改

建议阶段性迁移：

第一阶段：

```text
保留 JsonSessionStore 作为 session 元数据存储
messages 状态改由 checkpointer 保存
```

session JSON 可以只保存：

```json
{
  "id": "...",
  "created_at": "...",
  "updated_at": "...",
  "title": "...",
  "character_id": "nanami"
}
```

Graph 调用改成：

```python
config = {"configurable": {"thread_id": session["id"]}}
app.invoke({"question": question}, config=config)
```

第二阶段：

```text
从内存 checkpointer 迁移到 SQLite checkpointer
```

### 优先级

高。

这是 LangGraph 项目里最核心的框架化方向之一。

---

## 3. 长期记忆存储框架化

### 当前实现

相关文件：

```text
human_chat/memory_store.py
human_chat/memory_extractor.py
human_chat/storage/json_store.py
human_chat/graph.py
human_chat/cli.py
```

当前长期记忆保存在：

```text
data/memory/user_profile.json
```

由 `JsonMemoryStore` 读写，并通过 `prepare_context` 注入 prompt。

### 应该使用的框架能力

LangGraph 有 Store 概念，适合长期记忆。

长期记忆可以通过：

```text
namespace
key
value
```

组织为跨线程、跨会话的持久 JSON 文档。

工具和节点也可以通过运行时 store 读写长期记忆。

### 为什么要改

当前 JSON 长期记忆的问题：

```text
只有一个 user_profile.json，不适合多用户
缺少 namespace 和 key
难以支持多条 memory item 的独立更新
难以和工具调用、human-in-the-loop 结合
```

使用 LangGraph Store 后：

```text
可以按 user_id / project_id / character_id 分 namespace
可以把 memory item 单独存储
可以让工具节点读写 memory
可以和 checkpointer 一起形成完整状态体系
```

### 怎么改

建议先做 adapter，而不是一次性删除 JSON。

目标结构：

```text
MemoryRepository
  -> JsonMemoryRepository
  -> LangGraphStoreMemoryRepository
```

长期记忆可以组织为：

```text
namespace = ("users", user_id, "memory")
key = memory_id
value = {
  "category": "preference",
  "text": "...",
  "created_at": "...",
  "source": "confirmed_extraction"
}
```

Graph 的 `prepare_context` 从 Store 查询长期记忆并格式化。

### 优先级

中高。

应在短期会话 checkpointer 之后做，因为两者最好统一设计。

## 4. 人类确认流程框架化

### 当前实现

相关文件：

```text
human_chat/runtime.py
human_chat/cli.py
human_chat/memory_extractor.py
```

现在自动提取候选记忆后，`ChatRuntime.ask()` 返回：

```python
result["memory_candidates"]
```

然后 CLI 调用：

```python
_confirm_memory_candidates(...)
```

也就是说，人类确认逻辑在 Graph 外部。

### 应该使用的框架能力

LangGraph 支持 human-in-the-loop / interrupt。

适合这些场景：

```text
工具调用前确认
长期记忆写入前确认
危险操作前确认
让用户编辑模型生成内容
```

### 为什么要改

当前 CLI 外挂确认的问题：

```text
只适用于 CLI，不适用于未来 Web UI
确认流程不在 Graph 状态中
无法利用 checkpointer 暂停/恢复
后续如果工具调用也需要确认，会出现多套确认机制
```

使用 interrupt 后：

```text
Graph 可以暂停等待用户决定
CLI / Web UI 都可以处理同一种确认请求
确认结果回到 Graph 状态
记忆写入成为 Agent 流程的一部分
```

### 怎么改

建议将候选记忆提取移入 Graph：

```text
generate_reply
  -> extract_memory_candidates
  -> confirm_memory_update
  -> save_memory
  -> synthesize_speech
```

其中 `confirm_memory_update` 使用 interrupt。

确认动作支持：

```text
approve
reject
edit
```

### 优先级

中。

它依赖 checkpointer，所以应在短期会话框架化之后推进。

---

## 5. Graph 节点与状态模型框架化

### 当前实现

相关文件：

```text
human_chat/schemas.py
human_chat/graph.py
```

当前 `ChatState` 已经使用：

```python
Annotated[list[Any], add_messages]
```

这是正确方向。

但状态字段仍然存在一些临时痕迹：

```text
question
messages
memory_prompt
tool_request
tool_result
assistant_text
tts_text
tts_error
```

其中：

```text
assistant_text 和 tts_text 语义重复
tool_request/tool_result 是自定义工具实现遗留
question 独立于 messages，后续 checkpointer 化后可以重新设计
```

### 应该使用的框架能力

LangGraph 的状态模型应尽量围绕：

```text
messages
context
tool messages
structured runtime fields
```

工具调用结果可以进入 ToolMessage，而不是普通字符串字段。

### 为什么要改

清晰的状态模型能减少后续 bug。

当前状态字段如果继续增加，容易变成：

```text
每个功能加一个字段
节点之间依赖越来越隐式
某些字段只有部分节点使用
```

框架化后：

```text
对话上下文主要走 messages
工具结果走 ToolMessage
运行配置走 context / configurable
长期记忆作为节点读取的外部 store
TTS 作为输出副作用节点
```

### 怎么改

建议逐步整理：

```text
assistant_text 和 tts_text 合并为 assistant_text
tool_request/tool_result 迁移到 ToolNode / ToolMessage
question 最终也可以作为 HumanMessage 输入
memory_prompt 保持为节点内部派生值，必要时不持久化
```

### 优先级

高。

这应该和工具调用框架化、checkpointer 化一起做。

## 6. Runtime 层框架化

### 当前实现

相关文件：

```text
human_chat/runtime.py
```

当前 `ChatRuntime` 做了几件事：

```text
创建 graph app
维护 messages
保存 session
调用 memory extractor
返回结果给 CLI
```

它是我们为了从 CLI 中抽出核心逻辑而创建的运行层。

### 应该使用的框架能力

当 LangGraph checkpointer、Store、interrupt 接入后，Runtime 不应该再承担状态管理职责。

Runtime 更适合变成薄封装：

```text
准备 config
调用 graph
处理 streaming
把 framework events 转成 UI/CLI 可理解事件
```

### 为什么要改

如果 Runtime 继续管理 messages、memory_candidates、session 保存，就会和 Graph 框架能力重叠。

成熟结构应该是：

```text
Graph 管 Agent 状态和流程
Store / Checkpointer 管持久化
Runtime 管调用入口和事件转发
CLI / UI 管用户交互
```

### 怎么改

改造后 `ChatRuntime.ask()` 应类似：

```python
def ask(self, question: str):
    return self.app.invoke(
        {"messages": [HumanMessage(content=question)]},
        config={"configurable": {"thread_id": self.session_id}},
    )
```

如果使用 streaming：

```python
for event in self.app.stream(...):
    yield event
```

### 优先级

中高。

它依赖 checkpointer 和 Graph 状态整理。

## 7. 自动记忆提取框架化

### 当前实现

相关文件：

```text
human_chat/memory_extractor.py
human_chat/runtime.py
human_chat/cli.py
```

当前是在 `ChatRuntime.ask()` 结束后调用：

```python
extract_memory_candidates(...)
```

然后把候选结果交给 CLI 确认。

### 应该使用的框架能力

更合适的方式是将记忆提取作为 Graph 节点。

例如：

```text
generate_reply
  -> extract_memory_candidates
  -> interrupt_for_confirmation
  -> write_memory
```

### 为什么要改

自动记忆提取本质上是 Agent 流程的一部分，不应该藏在 Runtime 里。

放进 Graph 后：

```text
流程可视化更清晰
可以通过状态追踪候选记忆
可以接 human-in-the-loop
可以接 Store 写入
```

### 怎么改

新增 Graph 节点：

```text
extract_memory
confirm_memory
save_memory
```

状态中增加：

```python
memory_candidates: list[MemoryCandidate]
```

最终写入 LangGraph Store 或 MemoryRepository。

### 优先级

中。

建议在 checkpointer 和 interrupt 接入后做。

## 8. 会话选择与 thread_id 对齐

### 当前实现

相关文件：

```text
human_chat/cli.py
human_chat/session_store.py
```

CLI 现在可以：

```text
新建会话
继续最近会话
从列表选择会话
```

这是好的用户体验。

但会话 id 目前只用于 JSON 文件名，并没有和 LangGraph 的 thread_id 对齐。

### 应该使用的框架能力

LangGraph checkpointer 使用：

```text
thread_id
```

来区分不同会话状态。

### 为什么要改

如果 session_id 和 thread_id 分离，就会出现两套会话概念：

```text
CLI session
LangGraph thread
```

这会增加混乱。

### 怎么改

保持当前 session 选择体验，但让：

```text
session["id"] == thread_id
```

调用 graph 时：

```python
config={"configurable": {"thread_id": session["id"]}}
```

SessionStore 保存元数据，checkpointer 保存 Graph 状态。

### 优先级

高。

这是短期记忆框架化的必要配套。

## 9. 可观测性与调试框架化

### 当前实现

相关文件：

```text
human_chat/logging_config.py
```

当前主要是 Python logging。

这对本地开发有用，但对 Agent 调试还不够。

Agent 调试通常需要看：

```text
每个节点输入输出
工具调用参数
工具结果
模型消息
中断点
状态变化
```

### 应该使用的框架能力

LangChain / LangGraph 生态通常会配合：

```text
stream events
LangSmith tracing
节点级状态输出
```

### 为什么要改

当前错误排查主要依赖日志和 print。

随着工具、记忆、语音、UI 增多，如果没有框架级追踪，就很难回答：

```text
模型为什么调用了这个工具？
工具结果有没有进入 prompt？
长期记忆有没有被读取？
哪个节点失败了？
```

### 怎么改

建议逐步加入：

```text
Graph streaming
节点执行事件输出
可选 LangSmith 配置
CLI 调试模式
```

CLI 可以增加：

```text
/debug on
/debug off
```

调试模式下输出节点事件。

### 优先级

中。

建议在工具和 checkpointer 框架化后补上。

## 10. 仍然适合保留自定义实现的部分

不是所有内容都应该强行框架化。

以下部分保留自定义实现是合理的：

### 角色配置

相关文件：

```text
characters/*.yaml
human_chat/character.py
```

原因：

```text
这是产品层配置，不是 LangGraph 的核心职责
YAML 适合人类编辑角色提示词和 TTS 音色
```

### TTS 适配

相关文件：

```text
human_chat/tts.py
```

原因：

```text
GPT-SoVITS 是本地外部服务
需要项目专用 payload
属于 I/O 适配层
```

### STT 与录音

相关文件：

```text
human_chat/stt.py
human_chat/audio_recorder.py
human_chat/input_provider.py
```

原因：

```text
音频输入是产品交互层
LangGraph 不负责麦克风录音
```

### CLI 菜单

相关文件：

```text
human_chat/cli.py
```

原因：

```text
CLI 是当前产品入口
菜单、命令、输入模式切换属于用户交互层
```

但 CLI 应该逐渐变薄，不应该继续承担 Agent 状态管理。

---

## 建议改造顺序

综合风险和收益，建议按以下顺序推进：

```text
1. 工具定义与工具调用框架化
2. 短期会话记忆框架化：checkpointer + thread_id
3. Graph 状态模型整理
4. 长期记忆迁移到 LangGraph Store 或 Store adapter
5. 人类确认流程迁移到 interrupt
6. 自动记忆提取移入 Graph
7. Runtime 层变薄，专注事件转发
8. 加入 streaming / tracing / debug mode
```

这个顺序的原因：

```text
工具调用是当前最明显的重复实现，收益最大
checkpointer 是后续 interrupt 和长期记忆框架化的基础
状态模型整理要和工具、checkpointer 一起做
长期记忆 Store 和 human-in-the-loop 更适合在持久化能力稳定后做
```

---

## 总结

当前 HumanChat 已经具备 Agent 雏形，但很多基础能力还是“自己实现的一版”。

这些实现不是完全错误，它们帮助项目从脚本阶段走到了可用原型阶段。

但下一阶段要往成熟 Agent 项目发展，就应该逐步迁移到 LangChain / LangGraph 的标准能力：

```text
工具用 LangChain tools / ToolNode
短期记忆用 LangGraph checkpointer
长期记忆用 LangGraph Store
确认流程用 interrupt
状态流转交给 Graph
Runtime 和 CLI 逐渐变薄
```

这样项目后续接入 MCP、UI、流式输出、调试追踪、生产级持久化时，会更稳、更清晰，也更符合 LangGraph / LangChain 项目的开发方式。
