# 20 工具错误治理与可观测性修改记录

## 本次目标

本次是“完整工具调用框架化”的第四步：补齐工具错误治理、工具调用上限记录和基础可观测性。

前面几步已经完成：

```text
工具来源进入 ToolProvider
Graph 支持标准工具循环
tool_messages 被隔离为本轮 scratchpad
```

但商业项目还需要面对更现实的问题：

```text
工具可能失败
工具参数可能错误
工具调用可能重复
模型可能触发工具调用上限
排查问题时需要知道调用了哪些工具
```

所以本次重点不是增加新工具，而是让已有工具调用更稳定、更可观察。

## 修改了什么

### 1. 工具异常变成可控结果

在 `human_chat/tools.py` 中新增：

```python
def _run_project_tool(tool_name: str, callback):
    try:
        return callback()
    except Exception as exc:
        logger.warning("Project tool failed: %s", tool_name, exc_info=True)
        return f"[tool_error] {tool_name}: {exc}"
```

现有工具 wrapper 改为通过 `_run_project_tool` 执行：

```python
return _run_project_tool("read_project_file", lambda: read_project_file(PROJECT_ROOT, path))
```

这样工具内部出现异常时，不会直接让整轮 Graph 崩溃，而是返回一个明确的工具错误结果。

### 2. 新增工具调用事件状态

在 `ChatState` 中新增：

```python
tool_events: list[dict[str, Any]] = Field(default_factory=list)
tool_limit_reached: bool = False
```

`tool_events` 用来记录本轮工具调用摘要。

每条事件包含：

```text
round：第几轮工具调用
tool：工具名
arguments：工具参数
status：success / error / limit_reached
result_preview：工具结果预览
```

### 3. 工具执行后生成调用事件

在 `graph.py` 中新增：

```python
_build_tool_events
_tool_call_name
_tool_call_args
```

`execute_project_tools` 执行工具后，会根据模型请求的 tool calls 和 ToolNode 返回结果生成事件：

```python
"tool_events": [*state.tool_events, *_build_tool_events(state, tool_result_messages)]
```

这样每轮工具调用都会留下结构化摘要。

### 4. 工具调用达到上限时进入显式节点

新增节点：

```python
mark_tool_limit_reached
```

当模型还想继续调用工具，但已经达到 `MAX_TOOL_CALL_ROUNDS` 时，Graph 不再静默进入最终回复，而是先写入：

```python
tool_limit_reached = True
tool_events += limit_reached 事件
```

然后再进入 `generate_reply`。

这样调试时可以清楚知道：

```text
模型不是自然停止工具调用
而是被工具调用上限拦住了
```

### 5. 最终回复加入工具上限提示

如果本轮达到工具调用上限，`generate_reply` 会向模型补充：

```text
注意：本轮工具调用已达到上限，请基于已经获得的工具结果回答。
```

这样模型不会继续假设自己还能拿到更多工具结果，而是基于已有信息给出回答。

### 6. CLI debug 输出工具调用摘要

`/debug on` 模式下，现在会额外输出：

```text
tool_call_count
tool_events 数量
tool_limit_reached
每个工具事件的 tool / round / status
```

这不是完整 tracing 系统，但已经能帮助本地开发快速判断：

```text
模型是否调用了工具
调用了几轮
哪个工具失败了
是否触发调用上限
```

## 为什么这样做

### 1. 工具失败是生产常态

成熟项目不能假设工具永远成功。

工具可能因为很多原因失败：

```text
文件不存在
参数不合法
外部服务不可用
权限不足
编码格式不支持
网络超时
```

如果工具异常直接抛出，用户看到的就是整轮对话失败。

更好的方式是：

```text
工具失败被记录
失败信息作为 ToolMessage 返回给模型
模型基于失败信息给出解释或换一种方式尝试
```

### 2. 商业 Agent 必须有调用上限

工具循环如果没有上限，可能出现：

```text
无限调用
成本失控
响应时间过长
外部接口被打爆
用户体验变差
```

前一步已经引入 `MAX_TOOL_CALL_ROUNDS`，本次进一步把“达到上限”变成可观察状态。

这对排查问题很重要，因为最终回答质量不好时，我们需要知道原因是：

```text
工具结果不足
工具失败
模型没有调用工具
还是工具调用达到上限
```

### 3. 可观测性是商业项目的基本能力

Agent 系统比普通后端接口更难排查，因为一次回答中可能包含：

```text
模型判断
工具调用
工具结果
再次模型判断
最终回答
TTS 副作用
记忆提取
```

如果没有工具事件，就很难知道问题发生在哪里。

本次新增的 `tool_events` 是轻量版本的 tracing，为后续接入 LangSmith 或更完整的事件流打基础。

### 4. 错误返回要让模型能理解

工具失败时返回：

```text
[tool_error] tool_name: error
```

这个格式简单、稳定、模型容易理解。

后续可以继续扩展为更结构化的错误对象，例如：

```json
{
  "ok": false,
  "tool": "read_project_file",
  "error_type": "ValueError",
  "message": "不是文件：..."
}
```

当前先使用文本格式，是为了和 ToolNode / ToolMessage 保持兼容，同时不引入过多复杂度。

## 对商业项目的意义

### 1. 提升稳定性

工具失败不再轻易拖垮整轮对话。

### 2. 降低排查成本

开发者可以通过 debug 输出看到工具调用摘要。

### 3. 控制成本和风险

工具调用上限被显式记录，避免无限循环失控。

### 4. 为未来 MCP 工具打基础

MCP 工具比当前本地工具更复杂，也更容易遇到：

```text
连接失败
权限不足
远端服务报错
工具名冲突
返回内容过长
```

在接 MCP 前先建立错误治理和可观测性，是更稳的工程顺序。
