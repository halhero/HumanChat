# 28 长期记忆去分类化一致性修复修改记录

## 本次目标

本次修复的目标是协调第 23 步长期记忆数据模型结构化之后遗留的语义冲突。

第 23 步的目标是让长期记忆从：

```json
{
  "preferences": [],
  "facts": [],
  "notes": []
}
```

升级为：

```json
{
  "items": []
}
```

也就是说，长期记忆不应该再以 `preferences`、`facts`、`notes` 这类分类数组作为持久化结构。

但代码里仍然存在一个不一致点：

```python
class MemoryItem(BaseModel):
    category: str
    text: str
```

这导致实际保存长期记忆时，虽然不再使用顶层分类数组，但每条记忆仍然保存了 `category` 字段。

本次修复就是移除持久化长期记忆中的 `category` 字段，让长期记忆真正成为无分类的结构化记忆项。

## 修改了什么

### 1. `MemoryItem` 移除 `category`

修改前：

```python
class MemoryItem(BaseModel):
    id: str
    category: str
    text: str
    created_at: str
    updated_at: str
    source: str
    confidence: float | None
```

修改后：

```python
class MemoryItem(BaseModel):
    id: str
    text: str
    created_at: str
    updated_at: str
    source: str
    confidence: float | None
```

现在真正落盘的长期记忆项不再带分类字段。

### 2. 旧格式继续兼容读取

旧格式仍然可以读取：

```json
{
  "preferences": ["..."],
  "facts": ["..."],
  "notes": ["..."]
}
```

读取时会转换成：

```python
MemoryItem(text="...", source="legacy")
```

旧的 `items` 中如果已经存在 `category` 字段，Pydantic 会忽略这个额外字段，不影响读取。

### 3. 长期记忆 prompt 不再按分类分组

修改前：

```text
用户偏好：
- ...
长期事实：
- ...
备注：
- ...
```

修改后：

```text
长期记忆：
- ...
- ...
```

这样 prompt 展示和持久化模型保持一致。

### 4. `/memory` 命令去掉分类参数

修改前：

```text
/memory add preference 用户希望先看设计再改代码
/memory add fact HumanChat 使用 LangGraph
/memory add note 这是一条备注
/memory delete preference 1
```

修改后：

```text
/memory add 用户希望先看设计再改代码
/memory delete 1
```

删除操作现在按全局序号删除，不再按分类内序号删除。

### 5. 自动记忆提取不再要求分类

修改前，提取提示词要求模型输出：

```text
preference / fact / note
```

修改后，提取提示词明确要求：

```text
只返回候选记忆文本，不要给候选记忆分类。
```

`MemoryCandidate` 也同步移除 `category` 字段。

### 6. 自动提取确认保存时记录来源

CLI 中用户确认保存自动提取候选记忆时，现在会写入：

```python
source="extracted_confirmed"
```

用户主动 `/memory add` 仍然使用默认：

```python
source="manual"
```

这样虽然长期记忆不再分类，但仍然可以区分记忆来源。

### 7. 示例 JSON 同步去掉 `category`

`data/memory/user_profile.example.json` 中的每条 `items` 不再包含 `category` 字段。

## 为什么这样做

### 1. 避免“伪去分类化”

如果只是把顶层数组改成 `items`，但每个 item 里仍然保存 `category`，那实际上只是把分类从外层移动到了内层。

这和“长期记忆不再按 facts / notes 这类类别存储”的目标不一致。

本次修复后，长期记忆的持久化模型才真正变成：

```text
一组无分类 MemoryItem
```

### 2. 降低模型分类带来的噪声

自动记忆提取时，让模型判断 `preference` / `fact` / `note` 会引入额外不确定性。

对当前项目来说，最重要的是：

```text
这条信息是否值得长期记住
```

而不是：

```text
它到底属于 preference、fact 还是 note
```

所以提取阶段先只保留记忆文本更稳。

### 3. 删除逻辑更直接

去掉分类后，删除长期记忆可以直接按全局序号：

```text
/memory delete 1
```

不用再要求用户先判断这条记忆属于哪个分类。

### 4. 保留来源，不保留分类

长期记忆不再保存 category，但仍然保留 `source`。

这是必要的，因为来源属于审计信息，不是内容分类。

例如：

```text
manual
extracted_confirmed
legacy
default
example
```

这些信息能帮助后续判断记忆可信度和来源。

## 对成熟项目的意义

### 1. 数据模型更一致

长期记忆持久化不再同时存在：

```text
items 模型
category 分类字段
legacy 分类数组
```

当前准则变成：

```text
新数据只保存无分类 items
旧分类数据只作为兼容输入
```

### 2. 后续迁移更简单

未来接 LangGraph Store / SQLite 时，每条记忆只需要迁移：

```text
id
text
created_at
updated_at
source
confidence
```

不会再被旧分类体系绑定。

### 3. 用户操作更简单

用户不需要记住 category 参数。

### 4. 自动记忆提取更聚焦

模型只负责判断“是否值得记住”和“记忆文本是什么”，减少分类错误。
